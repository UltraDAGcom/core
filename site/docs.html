UltraDAG Documentation

Complete API reference, SDK guides, and node setup instructions

HTTP RPC API

The UltraDAG node exposes an HTTP RPC API on port 8080 by default. All endpoints return JSON and support CORS for web applications.

GET /status

Returns current network status including DAG round, vertices, and account count.

{
  "dag_round": 182,
  "dag_vertices": 728,
  "account_count": 17,
  "total_supply": "21000000.00000000",
  "last_finalized_round": 180,
  "network_id": "testnet",
  "uptime_seconds": 86400
}

GET /balance/:address

Query account balance and nonce for a given address.

{
  "address": "0x1234...",
  "balance": "1000.50000000",
  "nonce": 5,
  "staked_amount": "0.00000000",
  "rewards_earned": "12.34000000"
}

GET /round/:round

Get all vertices in a specific DAG round.

{
  "round": 182,
  "vertices": [
    {
      "hash": "0xabcd...",
      "validator": "0x1234...",
      "tx_count": 3,
      "reward": "0.00001000",
      "timestamp": "2026-03-12T17:00:00Z"
    }
  ],
  "is_finalized": true
}

GET /validators

List all active validators in the network.

{
  "validators": [
    {
      "address": "0x1234...",
      "stake": "10000.00000000",
      "commission_rate": "0.05",
      "uptime": 0.98,
      "rounds_participated": 180
    }
  ],
  "total_validators": 4,
  "required_validators": 3
}

POST /submit_tx

Submit a signed transaction to the mempool.

{
  "from": "0x1234...",
  "to": "0x5678...",
  "amount": "10.00000000",
  "fee": "0.00001000",
  "nonce": 5,
  "signature": "0xabcd...",
  "type": "transfer"
}

POST /stake

Submit a staking transaction to become a validator.

{
  "validator_pubkey": "0xabc123...",
  "amount": "10000.00000000",
  "commission_rate": "0.05",
  "signature": "0xdef456..."
}

Error Responses

All endpoints may return error responses with appropriate HTTP status codes:

{
  "error": "Invalid address format",
  "code": 400,
  "details": "Address must be a valid hex string"
}

SDK Quickstart

Install the UltraDAG SDK via npm:

npm install ultradag-sdk

Basic Usage

import { UltraDAG } from 'ultradag-sdk';

const client = new UltraDAG('http://localhost:8080');

// Get network status
const status = await client.getStatus();
console.log('Current round:', status.dag_round);

// Check balance
const balance = await client.getBalance(address);
console.log('Balance:', balance.balance);

// Send transaction
const tx = await client.sendTransaction({
  to: recipientAddress,
  amount: '10.0',
  privateKey: yourPrivateKey
});
console.log('Transaction submitted:', tx.hash);

Advanced Usage

// Create wallet from private key
const wallet = client.createWallet(privateKey);
console.log('Address:', wallet.address);

// Stake tokens to become validator
const stakeTx = await client.stake({
  amount: '10000.0',
  commissionRate: '0.05',
  privateKey: privateKey
});

// Monitor transaction confirmation
const confirmation = await client.waitForTransaction(tx.hash);
console.log('Confirmed in round:', confirmation.round);

// Get validator information
const validators = await client.getValidators();
const myValidator = validators.find(v => v.address === wallet.address);

// Stream network events
client.on('round', (round) => {
  console.log('New round:', round.number);
});

client.on('transaction', (tx) => {
  if (tx.to === wallet.address) {
    console.log('Received transaction:', tx.amount);
  }
});

Error Handling

try {
  const tx = await client.sendTransaction({
    to: recipientAddress,
    amount: '10.0',
    privateKey: privateKey
  });
} catch (error) {
  if (error.code === 'INSUFFICIENT_BALANCE') {
    console.log('Not enough funds for transaction');
  } else if (error.code === 'INVALID_NONCE') {
    console.log('Nonce mismatch, try refreshing account state');
  } else {
    console.error('Transaction failed:', error.message);
  }
}

Configuration

const client = new UltraDAG('http://localhost:8080', {
  timeout: 30000,        // Request timeout in ms
  retryAttempts: 3,     // Number of retry attempts
  retryDelay: 1000,      // Delay between retries
  apiKey: 'your-api-key', // Optional API key
  network: 'testnet'     // Network identifier
});

Running a Node

System Requirements

  • CPU: 2+ cores (4+ recommended for validators)
  • RAM: 4GB minimum (8GB+ recommended)
  • Storage: 50GB SSD (grows with network activity)
  • Network: Stable internet connection (1Mbps+)
  • OS: Linux (Ubuntu 20.04+, CentOS 8+), macOS 10.15+, Windows 10+

Installation

Binary Download (Recommended)

# Download latest release
curl -L https://github.com/UltraDAGcom/core/releases/latest/download/ultradag-node-linux-x64.tar.gz | tar xz

# Run the node
./ultradag-node --help

Build from Source

# Install Rust
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh

# Clone repository
git clone https://github.com/UltraDAGcom/core.git
cd core

# Build and run
cargo run --bin ultradag-node --release

Quick Start

Start a testnet node with default settings:

./ultradag-node --network testnet --data-dir ./ultradag-data

Configuration Options

./ultradag-node [OPTIONS]

OPTIONS:
    --network         Network to join [testnet|mainnet] [default: testnet]
    --data-dir           Data directory for state and DAG storage
    --port               HTTP RPC port [default: 8080]
    --p2p-port           P2P network port [default: 9000]
    --validators            Number of validators in the network [default: 4]
    --round-ms             Round duration in milliseconds [default: 5000]
    --prune-interval        Prune old DAG vertices every N rounds [default: 1000]
    --log-level        Log level [trace|debug|info|warn|error] [default: info]
    --enable-metrics          Enable Prometheus metrics on /metrics
    --bootstrap         Comma-separated list of bootstrap peers
    --private-key        Private key for validator (hex string)
    --help                    Print help information

Configuration File

Create a config.toml file for persistent configuration:

[network]
network = "testnet"
data_dir = "./ultradag-data"
port = 8080
p2p_port = 9000
bootstrap_peers = ["node1.ultradag.test:9000", "node2.ultradag.test:9000"]

[consensus]
validators = 4
round_ms = 5000
prune_interval = 1000

[logging]
level = "info"
enable_metrics = true

[validator]
private_key = "your_private_key_here"
commission_rate = "0.05"

Docker Setup

# Pull the image
docker pull ultradag/node:latest

# Run with default configuration
docker run -d \
  --name ultradag-node \
  -p 8080:8080 \
  -p 9000:9000 \
  -v ultradag-data:/data \
  ultradag/node:latest

# Run with custom config
docker run -d \
  --name ultradag-node \
  -p 8080:8080 \
  -p 9000:9000 \
  -v $(pwd)/config.toml:/config.toml \
  -v ultradag-data:/data \
  ultradag/node:latest --config /config.toml

Staking & Validation

To become a validator, you must stake at least 10,000 UDAG.

Stake Transaction

{
  "type": "stake",
  "amount": "10000.00000000",
  "validator_pubkey": "0x..."
}

Unstaking

Unstaking has a cooldown period of 2,016 rounds (~1 week at 5s rounds).

Governance

UltraDAG uses on-chain governance for protocol upgrades and parameter changes.

Proposal Types

  • Parameter Change - Modify network parameters (round time, fees, etc.)
  • Validator Set Update - Add or remove validators
  • Protocol Upgrade - Deploy new consensus rules

Architecture

UltraDAG consists of three main crates:

  • ultradag-coin - Core consensus, state, and crypto
  • ultradag-network - P2P networking and sync
  • ultradag-node - CLI binary and HTTP RPC server

Transaction Format

All transactions are Ed25519-signed with the following structure:

{
  "from": "Blake3(pubkey)",
  "to": "recipient_address",
  "amount": "satoshis",
  "fee": "satoshis",
  "nonce": "account_nonce",
  "signature": "Ed25519(NETWORK_ID || from || to || amount || fee || nonce)"
}

Checkpoints & Fast-Sync

Every 1,000 finalized rounds, validators produce a quorum-signed checkpoint.

Checkpoint Structure

{
  "round": 1000,
  "state_root": "0x...",
  "dag_tip": "0x...",
  "total_supply": "21000000.00000000",
  "signatures": ["0x...", "0x...", ...]
}

Troubleshooting

Node won't start

Check that the data directory is writable and ports are not in use. Verify system requirements are met.

Transactions not finalizing

Ensure at least ⌈2n/3⌉ validators are online and connected. Check network connectivity and firewall settings.

High memory usage

Enable pruning with --prune-interval 1000 to remove old DAG vertices. Consider increasing RAM or reducing data retention.

Sync issues

Check bootstrap peer addresses and ensure P2P port (9000) is accessible. Try resyncing from a recent checkpoint.

Validator rewards not received

Verify validator is in the active set and has sufficient uptime (>95%). Check commission rate settings.

Security Best Practices

Private Key Management

  • Store private keys offline in hardware wallets or secure enclaves
  • Use separate keys for staking and regular transactions
  • Regularly rotate validator keys (requires restaking)
  • Never share private keys or store them in version control

Node Security

  • Keep node software updated to latest version
  • Use firewall to restrict access to RPC port (8080)
  • Enable rate limiting on public endpoints
  • Monitor logs for suspicious activity
  • Use VPN or SSH tunnel for remote access

Network Security

  • Configure TLS/SSL for RPC endpoints in production
  • Use authenticated bootstrap peers
  • Implement DDoS protection for public nodes
  • Regular security audits of validator infrastructure

Common Attack Vectors

UltraDAG is designed to resist common blockchain attacks:

  • 51% Attack: Requires controlling >2/3 of validators
  • Long-Range Attacks: Prevented by checkpoint system
  • Eclipse Attacks: Mitigated by diverse peer connections
  • Sybil Attacks: Prevented by economic staking requirements

Performance & Optimization

Throughput Metrics

  • Transactions per Second: ~1,000 TPS (theoretical)
  • Block Finality: 5 seconds (one round)
  • Network Latency: <100ms for typical transactions
  • Storage Growth: ~1GB per 100K transactions

Optimization Tips

For Validators

  • Use SSD storage for faster I/O operations
  • Locate nodes close to network majority (low latency)
  • Enable pruning to control storage growth
  • Monitor CPU usage and optimize validator scheduling

For Applications

  • Batch transactions when possible
  • Use connection pooling for SDK clients
  • Implement local caching for frequently accessed data
  • Use WebSocket subscriptions for real-time updates
