HTTP RPC API

API Reference

Every UltraDAG node exposes a JSON HTTP API. The default RPC port is the P2P port + 1000 — so a node on :9333 serves its API at :10333. All responses include Access-Control-Allow-Origin: * headers.

The live testnet nodes are accessible at:

ultradag-node-1.fly.dev
ultradag-node-2.fly.dev
ultradag-node-3.fly.dev
ultradag-node-4.fly.dev

Units

All monetary values in the API use sats. 1 UDAG = 100,000,000 sats.

1 UDAG → 100,000,000 sats
1 sat → 0.00000001 UDAG

Infrastructure

Running a Node

UltraDAG nodes are distributed as a single self-contained binary (~2 MB). No runtime dependencies, no Docker required. Download, run, join the network.

ℹ

The binary is built deterministically via GitHub Actions and published to GitHub Releases on every push to main.

Quick Start

shell 
# Download pre-built binary (Linux x86_64)
curl -L https://github.com/UltraDAGcom/core/releases/latest/download/ultradag-node-linux-x86_64.tar.gz | tar xz
./ultradag-node --port 9333 --validate

# Build from source
git clone https://github.com/UltraDAGcom/core
cd core && cargo build --release
./target/release/ultradag-node --port 9333 --validate

CLI Flags

  • --portP2P listen port
    default: 9333
  • --rpc-portHTTP RPC port
    default: P2P + 1000
  • --seedSeed peer address (host:port). Can specify multiple.
  • --validateEnable block production
  • --round-msRound duration in milliseconds
    default: 5000
  • --validatorsExpected validator count (fixes quorum threshold)
  • --data-dirData directory for persistence
    default: ~/.ultradag/node
  • --pruning-depthRounds to keep before pruning
    default: 1000
  • --archiveDisable pruning, keep full history

Local Multi-Node Testnet

shell 
# Node 1 (bootstrap)
./ultradag-node --port 9333 --validate --validators 3

# Node 2
./ultradag-node --port 9334 --seed 127.0.0.1:9333 --validate --validators 3

# Node 3
./ultradag-node --port 9335 --seed 127.0.0.1:9333 --validate --validators 3

State Persistence

State is saved to --data-dir every 10 rounds and on graceful shutdown. Writes use atomic rename for crash safety. The write-ahead log (WAL) closes the gap between snapshots — finalized vertex batches are fsynced to wal.jsonl and replayed on restart.

FileContents
dag.jsonDAG vertices, tips, rounds, equivocation evidence
finality.jsonFinalized vertex hashes, validator set
state.jsonAccount balances, nonces, stake accounts
wal.jsonlWrite-ahead log for crash recovery
checkpoints/Accepted checkpoints (every 100 finalized rounds)
Validation

Staking & Validation

The top 21 stakers by amount become active validators at each epoch boundary. Validators produce DAG vertices, earn block rewards, and are slashed for equivocation.

Becoming a Validator

Run a node with --validate, fund your address with at least 10,000 UDAG, then stake via the API. The validator set updates every 210,000 rounds (~1 year).

shell 
curl -X POST https://ultradag-node-1.fly.dev/stake \
  -H "Content-Type: application/json" \
  -d '{
    "secret_key": "your_secret_key_hex",
    "amount": 1000000000000
  }'

Emission Schedule

Max supply is 21,000,000 UDAG. Initial reward is 50 UDAG per vertex, halving every 210,000 rounds. Rewards are distributed proportionally to stake.

⚠

Equivocation results in an immediate 50% stake burn and permanent validator ban. Ensure your node has a reliable connection before staking.

Architecture

Architecture Overview

UltraDAG is structured as a three-crate Rust workspace. Each layer has a single responsibility and a clean boundary.

LayerCratePurpose
0 ultradag-coin Ed25519 keys, DAG-BFT consensus, StateEngine, staking
1 ultradag-network TCP P2P: peer discovery, DAG vertex relay, state sync
2 ultradag-node Full node binary: validator loop + networking + HTTP RPC

DAG-BFT Consensus

UltraDAG is a pure DAG-driven ledger — there is no separate blockchain. The DAG is the ledger. Key properties:

PropertyDetail
LeaderlessAll validators produce vertices in parallel, no leader election
BFT finalityA vertex finalizes when >⅔ validators have descendants (O(1) via incremental tracking)
Optimistic responsivenessValidators produce immediately when 2f+1 previous-round vertices arrive
Equivocation preventionDuplicate validator+round vertices are rejected; equivocators are permanently banned
Deterministic orderingFinalized vertices ordered by (round, depth, hash) for state application

Cryptography

Signatures use Ed25519 via ed25519-dalek. Addresses are blake3(ed25519_public_key) — 32 bytes, hex-encoded. All signable bytes are prefixed with NETWORK_ID for domain separation.

✓

The consensus protocol has been formally verified using TLA+ across all 32.6 million possible executions of a 4-validator, 1-Byzantine network. Read the whitepaper →

DAG vertex
// Simplified vertex structure
struct DagVertex {
  // Identity
  hash:       [u8; 32],
  validator:  Address,
  round:      u64,
  timestamp:  i64,

  // DAG links (up to 64 parents)
  parents:    Vec<[u8; 32]>,

  // Payload
  transactions: Vec<Transaction>,
  reward:       u64,

  // Authentication
  signature:  [u8; 64],
}

// Finality: >2/3 validators with
// descendants in same round
fn is_finalized(v: &DagVertex) -> bool {
  let q = (2 * N + 2) / 3;
  v.descendant_count >= q
}

Signable Bytes Format

Transactions are signed over a deterministic byte representation:

format 
b"transfer" || from || to || amount || fee || nonce || memo_len || memo

All integers are little-endian u64. Type prefix prevents cross-type signature replay.

Nonce Management

Each address has a nonce counter starting at 0. Transactions must use current_nonce + 1. Check current nonce via GET /balance/:address. Mempool transactions increment the expected nonce, so pending transactions must be sequential.

Governance

Governance System

On-chain governance allows validators to propose and vote on protocol parameter changes. Proposals require minimum stake and go through a multi-phase lifecycle.

Proposal Lifecycle

1. Pending
Just created, voting not started
2. Active
Voting period open (default: 10,080 rounds)
3. Passed Pending
Passed, waiting for execution delay
4. Executed
Changes applied to runtime params
5. Rejected
Failed quorum or approval threshold
6. Expired
Voting period ended without quorum

Voting Requirements

Quorum
≥ 33% of total staked UDAG must vote
Approval
≥ 66% of votes must be "yes"
Min Stake
50 UDAG to create proposal
Voting Period
10,080 rounds (~14 hours at 5s)
Execution Delay
1,008 rounds (~1.4 hours) after passing

Proposal Types

ParameterChange — Modify runtime governance parameters (quorum threshold, voting period, execution delay, min stake, observer reward %, max active proposals).

TextProposal — Signal voting for off-chain coordination (no automatic execution).

Vote Weight

Vote weight equals staked amount at proposal creation. Addresses in unstake cooldown cannot vote. One address = one vote (no vote splitting).

Create Proposal
// Create parameter change proposal
const proposal = await fetch('/proposal', {
  method: 'POST',
  body: JSON.stringify({
    proposer_secret: "...",
    title: "Reduce min stake",
    description: "Lower barrier...",
    proposal_type: {
      ParameterChange: {
        param: "min_stake_to_propose",
        new_value: 25_00_000_000
      }
    }
  })
})

// Vote on proposal

  method: 'POST',
  body: JSON.stringify({
    voter_secret: "...",
    proposal_id: 1,
    vote: "yes" // or "no"
  })
})
Fast-Sync

Checkpoints & Fast-Sync

Checkpoints enable new nodes to sync from a recent snapshot instead of replaying the entire DAG history. Validators co-sign checkpoints every 100 finalized rounds.

Checkpoint Structure

round
Finalized round number
state_root
Blake3 hash of account state
dag_tip
Hash of latest finalized vertex
total_supply
Total UDAG in circulation
prev_checkpoint_hash
Hash of previous checkpoint (chain)
signatures
Map of validator → Ed25519 signature

Co-Signing Protocol

When a validator produces a checkpoint (every 100 finalized rounds):

  1. Compute state_root from current account balances
  2. Sign checkpoint with validator key
  3. Broadcast CheckpointProposal to all peers
  4. Peers verify state_root, sign if valid, send CheckpointSignature
  5. Collect signatures until quorum (⌈2n/3⌉) reached
  6. Save accepted checkpoint to checkpoints/ directory

Fast-Sync Process

New nodes joining the network:

  1. Send GetCheckpoint to bootstrap peer
  2. Receive checkpoint + suffix vertices + state snapshot
  3. Verify checkpoint chain back to genesis
  4. Verify quorum signatures (⌈2n/3⌉ validators)
  5. Load state snapshot into StateEngine
  6. Insert suffix vertices into DAG
  7. Begin normal consensus participation

Security

Checkpoints form a hash chain via prev_checkpoint_hash, linking back to hardcoded GENESIS_CHECKPOINT_HASH. Forged checkpoints are rejected (chain break detected). Eclipse attacks require rewriting from genesis (infeasible).

ℹ

Checkpoint pruning keeps only the 10 most recent checkpoints to limit disk usage. Older checkpoints are automatically deleted after new ones are produced.

Troubleshooting

Common Issues

Solutions to frequently encountered problems when running UltraDAG nodes.

High Finality Lag

Symptom: finality_lag > 10 rounds, transactions not confirming

Causes:

  • Network partition — validator can't reach 2/3 quorum
  • Clock skew — validator timestamps rejected by peers
  • Insufficient peers — isolated from network
  • Validator not producing — check --validate flag

Solutions:

  1. Check peer count: curl http://localhost:10333/peers
  2. Verify network connectivity to seed nodes
  3. Check system clock: timedatectl status
  4. Review logs for "equivocation" or "signature verification failed"
  5. Restart node with --seed pointing to known-good peers

Transaction Stuck in Mempool

Symptom: Transaction shows in /mempool but never confirms

Causes:

  • Incorrect nonce — must be exactly current_nonce + 1
  • Insufficient balance — amount + fee exceeds available balance
  • Fee too low — other transactions have higher priority
  • Signature invalid — pubkey doesn't match sender address

Solutions:

  1. Check current nonce: GET /balance/:address
  2. Verify balance covers amount + fee + pending tx costs
  3. Increase fee to at least 100 sats for priority
  4. Verify signature over correct signable_bytes format

Node Won't Start

Symptom: Process exits immediately or fails to bind port

Solutions:

  1. Check port availability: lsof -i :9333
  2. Verify data directory permissions: ls -la ~/.ultradag/node
  3. Check for corrupted state: try --clean-state (testnet only)
  4. Review logs in ~/.ultradag/node/ultradag.log
  5. Ensure sufficient disk space (min 1GB free)

Peer Connection Failures

Symptom: peer_count = 0, "connection refused" in logs

Solutions:

  1. Verify firewall allows TCP on P2P port (default 9333)
  2. Check seed node addresses are correct and reachable
  3. Try multiple seed nodes: --seed node1:9333 --seed node2:9333
  4. Verify DNS resolution if using hostnames
  5. Check for NAT/firewall blocking outbound connections

Memory Usage Growing

Symptom: RSS memory increases over time, eventual OOM

Solutions:

  1. Reduce pruning depth: --pruning-depth 500 (default 1000)
  2. Verify pruning is enabled (not --archive mode)
  3. Check for orphan buffer growth (indicates sync issues)
  4. Monitor checkpoint count (should stay ≤ 10)
  5. Restart node periodically if memory leak suspected
⚠

For production issues, always check /health/detailed first for component-level diagnostics. Enable debug logging with RUST_LOG=debug for detailed troubleshooting.
