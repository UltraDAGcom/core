HTTP RPC API

The UltraDAG node exposes an HTTP RPC API on port 10333 (P2P port + 1000). All endpoints return JSON and support CORS for web applications.

GET /status

Returns current network status including DAG round, vertices, and account count.

response — json 
{
  "dag_round": 14208,
  "dag_vertices": 56832,
  "account_count": 17,
  "total_supply": 2845600000000000,
  "last_finalized_round": 14206,
  "peer_count": 4,
  "mempool_count": 0
}

GET /balance/:address

Query account balance and nonce for a given address.

response — json 
{
  "address": "a1b2c3...",
  "balance_udag": "1000.50000000",
  "balance_sats": 100050000000,
  "nonce": 5
}

GET /round/:round

Get all vertices in a specific DAG round.

response — json 
{
  "round": 14208,
  "vertices": [
    {
      "hash": "a9f2c1...",
      "validator": "b7f3a1...",
      "tx_count": 3,
      "reward": 5000000000,
      "parent_count": 4
    }
  ]
}

GET /validators

List all active validators in the network.

response — json 
{
  "validators": [
    {
      "address": "b7f3a1...",
      "stake": 1000000000000,
      "is_active": true
    }
  ],
  "total_validators": 5,
  "total_staked": 5000000000000
}

POST /tx

Submit a signed transaction to the mempool. The node signs the transaction using the provided secret key.

request — json 
{
  "from_secret": "a1b2c3d4...",
  "to": "e5f6a7b8...",
  "amount": 1000000000,
  "fee": 10000
}

POST /stake

Submit a staking transaction to become a validator. Minimum stake: 10,000 UDAG.

request — json 
{
  "secret_key": "a1b2c3d4...",
  "amount": 1000000000000
}

Error Responses

All endpoints may return error responses with appropriate HTTP status codes:

error — json 
{
  "error": "Invalid address format"
}

SDK Quickstart

Four official SDKs with local Ed25519 keygen, Blake3 address derivation, and typed responses.

Installation

install — bash 
# Python
pip install -e sdk/python/

# JavaScript / TypeScript
cd sdk/javascript && npm install

# Rust — add to Cargo.toml
cargo add ultradag-sdk

# Go
go get github.com/ultradag/sdk-go/ultradag

Python

quickstart — python 
from ultradag import UltraDagClient, Keypair

client = UltraDagClient("https://ultradag-node-1.fly.dev")
keypair = Keypair.generate()

# Check network status
status = client.get_status()
print(f"Round: {status.dag_round}, Finalized: {status.last_finalized_round}")

# Get faucet funds (testnet)
client.faucet(keypair.address, 100_000_000)  # 1 UDAG

# Send transaction
client.send_tx(keypair.secret_key_hex, recipient, 50_000_000, fee=10_000)

JavaScript / TypeScript

quickstart — javascript 
import { UltraDagClient, Keypair } from 'ultradag';

const client = new UltraDagClient('https://ultradag-node-1.fly.dev');
const keypair = Keypair.generate();

// Check network status
const status = await client.getStatus();
console.log('Round:', status.dag_round);

// Check balance
const balance = await client.getBalance(keypair.address);
console.log('Balance:', balance);

// Send transaction
const tx = await client.sendTx({
  secretKey: keypair.secretKeyHex,
  to: recipientAddress,
  amount: 50_000_000,
  fee: 10_000
});

Rust

quickstart — rust 
use ultradag_sdk::{Client, Keypair};

let client = Client::new("https://ultradag-node-1.fly.dev");
let keypair = Keypair::generate();

// Check status
let status = client.get_status().await?;
println!("Round: {}", status.dag_round);

Go

quickstart — go 
import "github.com/ultradag/sdk-go/ultradag"

client := ultradag.NewClient("https://ultradag-node-1.fly.dev")
keypair := ultradag.GenerateKeypair()

// Check status

// Send transaction
tx, _ := client.SendTx(keypair, recipient, 50_000_000, 10_000)

Running a Node

System Requirements

  • CPU: Any (runs on $5/mo cloud instances)
  • RAM: 256MB minimum (bounded by pruning)
  • Storage: ~100MB after sync (bounded, not growing)
  • Binary: <2 MB release build
  • OS: Linux (x86_64), macOS (ARM64, x86_64)

Binary Download (Recommended)

install — bash 
# Download and extract (Linux)
curl -L https://github.com/UltraDAGcom/core/releases/latest/download/ultradag-node-linux-x86_64.tar.gz | tar xz

# Run node (auto-connects to testnet bootstrap nodes)
./ultradag-node-linux-x86_64 --validate

# Check status (RPC defaults to P2P port + 1000 = 10333)
curl http://localhost:10333/status

Build from Source

build — bash 
# Install Rust
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh

# Clone and build
git clone https://github.com/UltraDAGcom/core.git
cd core
cargo build --release

# Run validator
cargo run --release -p ultradag-node -- --validate

CLI Options

options — cli 
--port <PORT>           # P2P listen port [default: 9333]
--rpc-port <PORT>       # HTTP RPC port [default: P2P + 1000]
--seed <ADDR>           # Seed peer (host:port), can repeat
--validate              # Enable block production
--validators <N>       # Expected validator count
--round-ms <MS>        # Round duration [default: 5000]
--data-dir <PATH>      # Data directory [default: ~/.ultradag/node]
--pruning-depth <N>   # Rounds to keep [default: 1000]
--archive               # Disable pruning, keep full history
--pkey <HEX>           # Validator private key (64-char hex)
--auto-stake <UDAG>   # Auto-stake after startup+sync
--no-bootstrap          # Disable auto bootstrap connection
--skip-fast-sync        # Use local state only

Validator Onboarding

validator — bash 
# Full validator setup: bring your own key + auto-stake
./ultradag-node --validate \
  --pkey your_64char_hex_private_key \
  --auto-stake 10000

# Check validator status
curl http://localhost:10333/validators
curl http://localhost:10333/stake/your_address

Docker

docker — bash 
# Run with Docker
docker run -d \
  --name ultradag-node \
  -p 9333:9333 \
  -p 10333:10333 \
  -v ultradag-data:/data \
  -e VALIDATORS=5 \
  -e ROUND_MS=5000 \
  ultradag/node:latest

Staking & Validation

To become a validator, you must stake at least 10,000 UDAG.

Stake Transaction

{
  "type": "stake",
  "amount": "10000.00000000",
  "validator_pubkey": "0x..."
}

Unstaking

Unstaking has a cooldown period of 2,016 rounds (~1 week at 5s rounds).

Governance

UltraDAG uses on-chain governance for protocol upgrades and parameter changes.

Proposal Types

  • Parameter Change - Modify network parameters (round time, fees, etc.)
  • Validator Set Update - Add or remove validators
  • Protocol Upgrade - Deploy new consensus rules

Architecture

UltraDAG consists of three main crates:

  • ultradag-coin - Core consensus, state, and crypto
  • ultradag-network - P2P networking and sync
  • ultradag-node - CLI binary and HTTP RPC server

Transaction Format

All transactions are Ed25519-signed with the following structure:

{
  "from": "Blake3(pubkey)",
  "to": "recipient_address",
  "amount": "satoshis",
  "fee": "satoshis",
  "nonce": "account_nonce",
  "signature": "Ed25519(NETWORK_ID || from || to || amount || fee || nonce)"
}

Checkpoints & Fast-Sync

Every 1,000 finalized rounds, validators produce a quorum-signed checkpoint.

Checkpoint Structure

{
  "round": 1000,
  "state_root": "0x...",
  "dag_tip": "0x...",
  "total_supply": "21000000.00000000",
  "signatures": ["0x...", "0x...", ...]
}

Troubleshooting

Node won't start

Check that the data directory is writable and ports are not in use. Verify system requirements are met.

Transactions not finalizing

Ensure at least ⌈2n/3⌉ validators are online and connected. Check network connectivity and firewall settings.

High memory usage

Enable pruning with --prune-interval 1000 to remove old DAG vertices. Consider increasing RAM or reducing data retention.

Sync issues

Check bootstrap peer addresses and ensure P2P port (9000) is accessible. Try resyncing from a recent checkpoint.

Validator rewards not received

Verify validator is in the active set and has sufficient uptime (>95%). Check commission rate settings.

Security Best Practices

Private Key Management

  • Store private keys offline in hardware wallets or secure enclaves
  • Use separate keys for staking and regular transactions
  • Regularly rotate validator keys (requires restaking)
  • Never share private keys or store them in version control

Node Security

  • Keep node software updated to latest version
  • Use firewall to restrict access to RPC port (8080)
  • Enable rate limiting on public endpoints
  • Monitor logs for suspicious activity
  • Use VPN or SSH tunnel for remote access

Network Security

  • Configure TLS/SSL for RPC endpoints in production
  • Use authenticated bootstrap peers
  • Implement DDoS protection for public nodes
  • Regular security audits of validator infrastructure

Common Attack Vectors

UltraDAG is designed to resist common blockchain attacks:

  • 51% Attack: Requires controlling >2/3 of validators
  • Long-Range Attacks: Prevented by checkpoint system
  • Eclipse Attacks: Mitigated by diverse peer connections
  • Sybil Attacks: Prevented by economic staking requirements

Performance & Optimization

Throughput Metrics

  • Transactions per Second: ~1,000 TPS (theoretical)
  • Block Finality: 5 seconds (one round)
  • Network Latency: <100ms for typical transactions
  • Storage Growth: ~1GB per 100K transactions

Optimization Tips

For Validators

  • Use SSD storage for faster I/O operations
  • Locate nodes close to network majority (low latency)
  • Enable pruning to control storage growth
  • Monitor CPU usage and optimize validator scheduling

For Applications

  • Batch transactions when possible
  • Use connection pooling for SDK clients
  • Implement local caching for frequently accessed data
  • Use WebSocket subscriptions for real-time updates

Benchmarking

Run built-in benchmarks to test your setup:

# Network latency test
./ultradag-node benchmark --latency-test --nodes 4

# Storage performance test
./ultradag-node benchmark --storage-test --data-dir ./test-data
