HTTP RPC API Reference

Every UltraDAG node exposes an HTTP JSON API. Default port is P2P port + 1000 (e.g., P2P 9333 → RPC 10333). All responses are JSON with CORS headers enabled.

Read Endpoints

EndpointMethodDescription
/health GET Health check. Returns {"status":"ok"}
/status GET Network overview: DAG round, finalized count, total supply, peers, mempool size, validators, staking info
/balance/:address GET Balance (sats + UDAG) and nonce for an address
/round/:round GET All DAG vertices in a round: hash, validator, reward, tx count, parent count
/mempool GET Pending transactions (top 100 by fee)
/keygen GET Generate new Ed25519 keypair (secret_key + address)
/peers GET Connected peers and bootstrap node status
/validators GET Active validators with stake amounts
/stake/:address GET Stake info: staked amount, unlock round, active validator status
/proposals GET Governance proposals list
/governance/config GET Governance configuration parameters

Write Endpoints

EndpointMethodBodyDescription
/tx POST {"from_secret", "to", "amount", "fee"} Submit signed transaction. Amount and fee in sats.
/faucet POST {"address", "amount"} Testnet faucet. Creates real signed tx from faucet keypair.
/stake POST {"secret_key", "amount"} Stake UDAG as validator. Amount in sats.
/unstake POST {"secret_key"} Begin unstake cooldown (2,016 rounds).
/proposal POST {"proposer_secret", "title", "description", "proposal_type", ...} Submit governance proposal.
/vote POST {"voter_secret", "proposal_id", "vote"} Vote on governance proposal.

Example: Check Balance

curl https://ultradag-node-1.fly.dev/balance/abc123...

# Response:
{
  "address": "abc123...",
  "balance": 10000000000,
  "nonce": 0,
  "balance_tdag": 100.0
}

Example: Send Transaction

curl -X POST https://ultradag-node-1.fly.dev/tx \
  -H "Content-Type: application/json" \
  -d '{
    "from_secret": "your_secret_key_hex",
    "to": "recipient_address_hex",
    "amount": 1000000000,
    "fee": 100000
  }'

Units

All monetary values in the API are in sats (1 UDAG = 100,000,000 sats). Convert with:

  • UDAG to sats: multiply by 100,000,000
  • Sats to UDAG: divide by 100,000,000

SDK Quickstart

Official SDKs for Python, JavaScript/TypeScript, Rust, and Go. Each provides local Ed25519 key generation + HTTP RPC client.

Python

pip install ultradag

from ultradag import UltraDagClient, Keypair

client = UltraDagClient("https://ultradag-node-1.fly.dev")
kp = Keypair.generate()
print(f"Address: {kp.address}")

# Check balance
bal = client.get_balance(kp.address)
print(f"Balance: {bal.balance} sats")

# Get testnet tokens
client.faucet(kp.address, 100 * 100_000_000)

JavaScript / TypeScript

npm install ultradag

import { UltraDagClient, Keypair } from 'ultradag';

const client = new UltraDagClient({ baseUrl: 'https://ultradag-node-1.fly.dev' });
const kp = await Keypair.generate();

const status = await client.getStatus();
console.log(`DAG round: ${status.dag_round}`);

Rust

# Cargo.toml
[dependencies]
ultradag-sdk = { path = "sdk/rust" }

use ultradag_sdk::{UltraDagClient, Keypair};

let client = UltraDagClient::new("https://ultradag-node-1.fly.dev");
let kp = Keypair::generate();
let bal = client.balance(&kp.address_hex())?;
println!("Balance: {} sats", bal.balance);

Go

import "github.com/ultradag/sdk-go/ultradag"

client := ultradag.NewClient("https://ultradag-node-1.fly.dev")
kp, _ := ultradag.GenerateKeypair()

status, _ := client.Status()
fmt.Printf("DAG round: %d\n", status.DagRound)

Full SDK source and tests: github.com/UltraDAGcom/core/tree/main/sdk

Running a Node

Quick Start

# Download and run (Linux x86_64)
curl -L https://github.com/UltraDAGcom/core/releases/latest/download/ultradag-node-linux-x86_64.tar.gz | tar xz
./ultradag-node --port 9333 --validate

# Or build from source
git clone https://github.com/UltraDAGcom/core
cd core && cargo build --release
./target/release/ultradag-node --port 9333 --validate

State Persistence

The node persists its state to --data-dir (default ~/.ultradag/node):

  • dag.json — DAG vertices, tips, rounds, equivocation evidence
  • finality.json — Finalized vertex hashes, validator set
  • state.json — Account balances, nonces, stake accounts
  • mempool.json — Pending transactions
  • checkpoints/ — Accepted checkpoints (every 1000 rounds)

State is saved every 10 rounds and on graceful shutdown (SIGTERM/SIGINT). Writes are crash-safe via atomic rename.

Staking & Validation

Becoming a Validator

  1. Run a node with --validate
  2. Fund your address with at least 10,000 UDAG (minimum stake)
  3. Stake via the API: POST /stake with your secret key and amount
  4. Top 21 stakers by amount become active validators at each epoch boundary

Staking Parameters

ParameterValueDescription
Minimum stake10,000 UDAGMinimum to be eligible for active validator set
Max validators21Top 21 stakers are active (odd for clean BFT quorum)
Unstake cooldown2,016 rounds~1 week at 5s rounds before funds unlock
Epoch length210,000 roundsValidator set recalculated at each epoch boundary
Observer reward20%Staked but not in active set earn 20% of normal reward
Slashing50%Stake burned on equivocation (deflationary)

Reward Distribution

Block rewards are distributed proportionally to stake. Each validator's reward = block_reward × (own_stake / total_stake). Before any staking is active, each vertex gets the full block reward.

Emission Schedule

  • Max supply: 21,000,000 UDAG
  • Initial reward: 50 UDAG per vertex (each validator earns 50 UDAG per block produced)
  • Halving: every 210,000 rounds
  • Genesis allocations: 1M UDAG faucet (testnet) + 1.05M UDAG developer (5%)

Architecture Overview

Three-Crate Design

LayerCratePurpose
0 — Coinultradag-coinEd25519 keys, DAG-BFT consensus, StateEngine, staking
1 — Networkultradag-networkTCP P2P: peer discovery, DAG vertex relay, state sync
2 — Nodeultradag-nodeFull node binary: validator loop + networking + HTTP RPC

DAG-BFT Consensus

UltraDAG is a pure DAG-driven ledger. There is no separate blockchain — the DAG is the ledger. Key properties:

  • Leaderless — all validators produce vertices in parallel, no leader election
  • BFT finality — a vertex finalizes when >2/3 validators have descendants of it (O(1) check via incremental tracking)
  • Optimistic responsiveness — validators produce immediately when 2f+1 previous-round vertices arrive
  • Equivocation prevention — duplicate validator+round vertices are rejected; equivocators are permanently banned
  • Deterministic ordering — finalized vertices ordered by (round, depth, hash) for state application
  • Parent finality guarantee — vertices only finalize after all parents are finalized

Cryptography

  • Signatures: Ed25519 (ed25519-dalek)
  • Addresses: blake3(ed25519_public_key) — 32 bytes, hex-encoded
  • Domain separation: all signable bytes prefixed with NETWORK_ID
