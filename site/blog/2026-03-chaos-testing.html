Testing March 2026

Chaos Engineering: Breaking UltraDAG to Prove It Works

Most blockchain projects test the happy path. Nodes start cleanly, network connections are stable, messages arrive in order. Everything works because nothing goes wrong.

That's not how production works.

In production, nodes crash mid-consensus. Network cables get unplugged. Packets get dropped. Clocks drift. Partitions happen. And when they do, the question isn't whether your whitepaper claims Byzantine fault tolerance—it's whether your code actually recovers.

We built a Jepsen-style chaos testing framework to answer that question with data instead of claims.

The Framework

Our fault injection infrastructure systematically injects four categories of faults into a running UltraDAG network:

Network partitions. Split-brain scenarios (2-2 splits), node isolation, minority/majority partitions (1/3 vs 2/3). We partition the network, let it run for a configurable duration, then heal the partition and verify all nodes converge to the same state.

Clock skew. Nodes don't have synchronized clocks in the real world. We inject time drift—some nodes run 30 seconds ahead, others 30 seconds behind. We test that consensus continues and that vertices with absurdly future timestamps (10 minutes ahead) get rejected.

Message chaos. Random delays up to 2 seconds. Message reordering. Packet drops at configurable rates (10-15%). We verify that consensus progresses despite unreliable message delivery.

Crash-restart. Kill a node mid-round. Restart it. Kill it again. We test single crashes, repeated crash-restart cycles, and simultaneous crashes of multiple nodes (as long as fewer than 1/3 crash, the network should continue).

The Tests

We wrote 14 long-running Jepsen tests that combine these faults in increasingly hostile scenarios. Each test runs for 10-30 seconds of simulated network time, injects faults, and then checks invariants.

The invariant checkers verify:

  • Finality safety — No two nodes finalize conflicting vertices at the same round
  • Finality irreversibility — Once finalized, a vertex never gets un-finalized
  • Supply consistency — All nodes agree on total supply after convergence
  • Double-spend prevention — No address spends the same funds twice

If any invariant is violated, the test fails and prints a detailed report showing exactly what broke.

Published Results

Here's what we learned by systematically breaking UltraDAG:

Node Crash Recovery

  • Killed node 0 at round 847
  • Network continued with 3 of 4 validators
  • Node 0 restarted and rejoined
  • Full recovery in 3 rounds (rounds 848-850)
  • Zero finality violations

Split-Brain Partition

  • 2-2 network split for 10 seconds
  • Neither partition could finalize (no 3/4 quorum)
  • Partition healed after 10 seconds
  • All 4 nodes converged to identical state
  • Finality resumed immediately after heal

Message Drop Resilience

  • 10% sustained packet loss for 20 seconds
  • Finality lag increased from 2 to 5 rounds during chaos
  • Consensus continued throughout
  • Lag returned to 2 rounds after packet loss cleared
  • No state divergence between nodes

Extreme Chaos Scenario

  • Phase 1: Network partition + 60-second clock skew
  • Phase 2: Partition healed, 1-second message delays + reordering enabled
  • Phase 3: Node crash during active consensus
  • Phase 4: 15% packet drops + 2-second delays simultaneously
  • Result: Zero finality violations. Network recovered.

What This Proves

Chaos testing doesn't prove correctness—formal verification does that. But it proves resilience. It proves that when things go wrong in production (and they will), the network recovers instead of diverging into an inconsistent state.

The specific numbers matter. "Killed node 0 at round 847, recovered in 3 rounds" is more credible than "our consensus is Byzantine fault tolerant" because it's a measurement, not a claim.

Every test is reproducible. The fault injection framework is part of the codebase. Anyone can run cargo test --test jepsen_tests -- --ignored and verify the same results.

Why This Matters for Mainnet

Mainnet will experience all of these faults. Validators will crash. Network cables will get unplugged. Cloud providers will have routing issues. Clocks will drift.

The question isn't whether faults happen—it's whether the network survives them. Chaos testing gives us confidence that it does.

We're publishing these results because resilience claims should be backed by data. If your consensus algorithm can't survive a Jepsen test, it's not ready for production.

Run the tests yourself:

cargo test --test jepsen_tests -- --ignored

Full test suite: crates/ultradag-network/tests/jepsen_tests.rs

What's Next

We're expanding the chaos test suite with longer-duration tests (hours instead of seconds) and more complex failure scenarios. We're also adding performance benchmarks under chaos conditions to measure how much throughput degrades when the network is under stress.

Chaos testing is never finished. Every new feature gets chaos tests. Every bug that makes it to testnet gets a regression test that reproduces the fault.

Breaking things systematically is how you build systems that don't break in production.
